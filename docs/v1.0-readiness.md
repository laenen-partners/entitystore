# EntityStore v1.0 Readiness Assessment

**Date:** 2026-03-24
**Author:** Pascal Laenen

---

## Executive Summary

EntityStore is functionally complete. The core pipeline (Extract → Match → Store → Query) works, the codegen eliminates boilerplate, and the test coverage is solid. What's missing for v1.0 is hardening: an interface for testability, input validation, safety caps on unbounded queries, and two missing database indexes.

This document covers findings from a full codebase audit, prioritized recommendations, and an analysis of advanced matching techniques (bloom filters, LSH, MinHash) for future consideration.

## Must-Fix Before v1.0

### 1. No interface for EntityStore / ScopedStore

**Problem:** Consumers cannot mock EntityStore in unit tests without defining their own interface. Every service that uses entitystore has to create a local interface that mirrors the methods they use. This is the single biggest DX gap for adopters.

**Impact:** Every consuming service duplicates interface definitions. When EntityStore adds a method, downstream interfaces may break or drift.

**Fix:** Define an `EntityStorer` interface in the root package that both `EntityStore` and `ScopedStore` satisfy. Consumers type their dependencies as `entitystore.EntityStorer`.

```go
type EntityStorer interface {
    GetEntity(ctx context.Context, id string) (StoredEntity, error)
    FindByAnchors(ctx context.Context, entityType string, anchors []AnchorQuery, filter *QueryFilter) ([]StoredEntity, error)
    FindByTokens(ctx context.Context, entityType string, tokens []string, limit int, filter *QueryFilter) ([]StoredEntity, error)
    FindByEmbedding(ctx context.Context, entityType string, vec []float32, topK int, filter *QueryFilter) ([]StoredEntity, error)
    BatchWrite(ctx context.Context, ops []BatchWriteOp) ([]BatchWriteResult, error)
    Traverse(ctx context.Context, entityID string, opts *TraverseOpts) ([]TraverseResult, error)
    // ... remaining methods
}
```

**Effort:** Small. Define the interface, verify both types satisfy it with compile-time checks.

### 2. Unbounded relation queries

**Problem:** Four relation queries have no LIMIT clause:

| Query | SQL file | Risk |
|---|---|---|
| `GetRelationsFromEntity` | relations.sql | Entity with 100K+ outbound relations returns all rows |
| `GetRelationsToEntity` | relations.sql | Same for inbound |
| `GetRelationsByType` | relations.sql | Returns all relations of a type globally |
| `ConnectedEntities` | store.go | Runs two unbounded queries, dedupes in Go |

An entity that accumulates relations over time (e.g., a company with 50K employees) will cause these queries to return massive result sets, consuming memory and degrading latency.

**Fix:** Add pagination (pageSize + cursor) to relation queries, matching the pattern already used by `FindConnectedByType`. Default limit of 1000. `ConnectedEntities` should merge into a single `UNION ALL` query with a LIMIT.

**Effort:** Small. The pagination pattern is already established in the codebase.

---

### 3. No input validation on tags, relation types, entity types

**Problem:** All string inputs are passed directly to the database without validation:

- **Tags:** No length limit. A caller can add a 10MB string as a tag, or millions of tags to a single entity. The GIN index on `tags` will degrade with pathological inputs.
- **Relation types:** No format validation. Empty strings, whitespace-only strings, and extremely long strings are all accepted.
- **Entity types:** Derived from `proto.MessageName()` for writes (safe), but read queries accept any string for entityType filtering.

**Fix:** Add validation in the store layer:
- Tags: max 255 chars per tag, max 100 tags per entity, reject empty strings
- Relation types: max 255 chars, non-empty, valid identifier characters
- Reject zero-value inputs where they don't make sense (empty entityID, empty relation type)

**Effort:** Small. Validation at the top of each method, before database calls.

---

### 4. Missing database indexes

**Problem:** Two query patterns lack supporting indexes:

| Query | Filter columns | Existing index | Missing |
|---|---|---|---|
| `FindConnectedByTypeInbound` | `target_id, relation_type` | `idx_rel_target` (target_id only) | Composite `(target_id, relation_type)` |
| `GetRelationsForSource` | `source_urn` | None | `(source_urn)` |

The inbound query is used by `FindConnectedByType` and `Traverse` — both hot paths. Without the composite index, PostgreSQL does an index scan on `target_id` then filters `relation_type` in-memory.

**Fix:** Add migration 3:

```sql
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_rel_target_type
    ON entity_relations (target_id, relation_type);
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_rel_source_urn
    ON entity_relations (source_urn) WHERE source_urn IS NOT NULL;
```

**Effort:** Small. One migration file, no application code changes.

---

### 5. No BatchWrite operation count cap

**Problem:** `BatchWrite` accepts a `[]BatchWriteOp` with no size limit. A caller passing 100K operations creates a single massive transaction that:
- Holds locks for the entire duration
- Accumulates WAL entries proportional to the batch size
- Can hit PostgreSQL's `max_locks_per_transaction` limit (default 64 per table, 6400 total)
- If it fails, the entire batch rolls back — wasting all work

**Fix:** Add a configurable max batch size. Default 1000 ops. Return a clear error if exceeded. Callers that need larger batches can chunk explicitly.

**Effort:** Small. One validation check at the top of `BatchWrite`.

## Should-Fix (High Value, Low Effort)

### 6. Add `GetByAnchor` convenience method

`FindByAnchors` returns `[]StoredEntity` because it supports multiple anchor queries. But the most common pattern is looking up a single entity by a single anchor — 95% of callers do:

```go
matches, _ := es.FindByAnchors(ctx, entityType, []AnchorQuery{{Field: "email", Value: email}}, nil)
if len(matches) == 0 { /* not found */ }
entity := matches[0]
```

Add a convenience method:

```go
func (es *EntityStore) GetByAnchor(ctx context.Context, entityType, field, value string, filter *QueryFilter) (StoredEntity, error)
```

Returns the first match or a well-defined "not found" error.

---

### 7. Auto-set ProvenanceEntry.ExtractedAt

Every write requires a `ProvenanceEntry` with `ExtractedAt` set. Callers always set it to `time.Now()`. The store should default `ExtractedAt` to the current time when the zero value is provided, reducing boilerplate:

```go
// Before:
store.WithProvenance(matching.ProvenanceEntry{
    SourceURN: urn, ModelID: model, ExtractedAt: time.Now(), MatchMethod: "create",
})

// After (ExtractedAt auto-filled):
store.WithProvenance(matching.ProvenanceEntry{
    SourceURN: urn, ModelID: model, MatchMethod: "create",
})
```

---

### 8. Structured logging (slog)

CLAUDE.md states "use slog for structured logging" but zero slog calls exist in the codebase. For a v1.0 library, consumers expect to be able to trace operations. At minimum:

- **Debug:** Traverse depth/result count, BatchWrite op count, FindByAnchors match count
- **Warn:** Near-limit operations (batch size > 500, traverse result > 80% of max)
- **Error:** Already handled by returning errors, but context logging helps

Implementation should accept `*slog.Logger` via an option (`WithLogger`) and default to `slog.Default()`. Operations should log only on completion (not entry) to keep log volume manageable.

## Advanced Matching Techniques: Bloom Filters and Alternatives

### Current matching pipeline

The entity resolution pipeline works in three stages:

```
1. Anchor lookup     → O(1) exact match via indexed table      → short-circuit if found
2. Candidate retrieval → Token overlap (GIN) + Embedding (HNSW) → top-K candidates
3. Field scoring      → Jaro-Winkler, Levenshtein, etc.        → weighted composite score
```

The question is whether bloom filters or other probabilistic structures could improve stage 2 (candidate retrieval) or add a new pre-filtering stage.

### Bloom filters

A bloom filter is a space-efficient probabilistic set: it can tell you "definitely not in the set" (no false negatives) or "probably in the set" (false positives possible).

**Where it could theoretically help:**

- **Pre-filter before anchor lookup:** Before hitting the database, check a bloom filter keyed on anchor values to skip entities that definitely don't exist. This saves a database round-trip for new entities.
- **Duplicate detection in batch ingestion:** When ingesting thousands of entities from a document, check a bloom filter to identify likely-new entities vs. likely-existing ones, routing them to different processing paths.

**Why it doesn't help EntityStore today:**

1. **Anchor lookups are already O(1).** The `entity_anchors` table has a B-tree index on `(entity_type, anchor_field, normalized_value)`. A bloom filter would add complexity for a lookup that's already a single index scan. The database is the source of truth — a bloom filter is an approximation of what the database already does fast.

2. **False positives are expensive.** A bloom filter that says "maybe exists" still requires a database query to confirm. For entity resolution, a false positive means running the full scoring pipeline on a non-match. The current anchor short-circuit (confidence 1.0 on match) is already the optimal path.

3. **Maintenance cost.** The bloom filter must be kept in sync with the database. Every entity create/delete must update the filter. In a multi-process deployment, the filter must be shared (Redis, shared memory) or rebuilt on startup. This adds operational complexity for marginal benefit.

4. **Cardinality mismatch.** Bloom filters shine when the set is large and queries are frequent (e.g., checking if a URL was visited in a web crawler with billions of URLs). EntityStore's anchor table is already indexed and typically has thousands to low millions of entries — well within PostgreSQL's index performance range.

**Verdict:** Bloom filters are not useful for EntityStore's current architecture. The anchor index already provides O(1) lookups with zero false positives.

### Locality-Sensitive Hashing (LSH)

LSH hashes similar items to the same bucket with high probability. It's used for approximate nearest-neighbor search in high-dimensional spaces.

**Where it could help:**

- **Pre-filtering for embedding search:** Before running the HNSW index query, use LSH to quickly identify candidate buckets, reducing the search space.
- **Cross-entity-type similarity:** Find similar entities across types without running separate embedding queries per type.

**Why it doesn't help today:**

- **HNSW already solves this.** The HNSW index in pgvector is itself an approximate nearest-neighbor structure. It provides sub-linear search time with tunable accuracy (via `ef_search`). Adding LSH on top would be redundant — you'd be approximating an approximation.
- **pgvector handles the hot path.** The embedding query runs entirely in PostgreSQL. Adding a Go-side LSH layer would require loading embeddings into memory, defeating the purpose of database-side search.

**When LSH would make sense:** If EntityStore moved to a sharded architecture where embeddings are spread across multiple databases, LSH could serve as a routing layer to identify which shard(s) to query. This is not a current or near-term need.

**Verdict:** Not useful while pgvector's HNSW index handles embedding search.

### MinHash for token similarity

MinHash estimates Jaccard similarity between sets without comparing every element. It's used for document deduplication at scale (billions of documents).

**Where it could help:**

- **Token-based candidate retrieval:** Instead of GIN index overlap queries, compute MinHash signatures for each entity's token set and compare signatures for O(1) similarity estimation.
- **Near-duplicate detection:** Quickly identify entity pairs that are likely duplicates based on token overlap.

**Current approach (GIN index):** EntityStore stores tokens in a `text[]` column with a GIN index. `FindByTokenOverlap` uses the `&&` (overlap) operator to find entities sharing at least one token, then ranks by overlap count. This is already efficient for the typical entity resolution case.

**Trade-off analysis:**

| Dimension | GIN index (current) | MinHash |
|---|---|---|
| Storage | Tokens stored directly | Tokens + MinHash signatures |
| Query speed | GIN scan, O(n) where n = matching entities | O(1) signature comparison |
| Accuracy | Exact overlap count | Approximate Jaccard similarity |
| Maintenance | Updated on write via SQLC | Signatures must be recomputed on token change |
| Scale threshold | Efficient up to ~10M entities per type | Efficient at 100M+ |

**When MinHash would make sense:** If a single entity type grows beyond 10 million entities and token-based candidate retrieval becomes a bottleneck. At that scale, the GIN index scan for `tokens && $input_tokens` starts to slow down because it must examine all matching posting lists. MinHash would allow pre-filtering to a smaller candidate set.

**Verdict:** Not needed now. Keep in mind for scale-out scenarios. If adopted, implement as a separate `matching/minhash` package that computes signatures on write and stores them in an additional column.

### Phonetic encoding (Soundex, Metaphone, Double Metaphone)

Encodes words by their pronunciation. "Smith" and "Smyth" produce the same code. Useful for name matching in multilingual contexts.

**Where it could help:**

- **Name deduplication:** "John Smith" vs. "Jon Smyth" would match on phonetic encoding even though Jaro-Winkler gives a moderate score.
- **Cross-language names:** "Mohammed" / "Muhammad" / "Mohamed" share the same Double Metaphone code.

**Current approach:** Jaro-Winkler similarity on normalized names. This handles typos and minor variations well but misses phonetic equivalences.

**Trade-off:** Phonetic encoding is a complement to Jaro-Winkler, not a replacement. It could be added as an additional normalizer in proto annotations:

```protobuf
string full_name = 2 [(entitystore.v1.field) = {
    normalizer: NORMALIZER_DOUBLE_METAPHONE
    // or as an additional comparison dimension
}];
```

**Verdict:** Worth considering for v1.1 if name matching accuracy is a pain point. Implementation is ~50 lines of Go for Double Metaphone, plus a new normalizer enum value and codegen support.

### Summary of techniques

| Technique | Useful for EntityStore? | When? |
|---|---|---|
| Bloom filter | No | N/A — anchor index already O(1) |
| LSH | No | Only if sharded across multiple databases |
| MinHash | Not yet | When a single entity type exceeds ~10M rows and token retrieval slows |
| Double Metaphone | Possibly | v1.1+ if name matching accuracy is a reported issue |
| HNSW (current) | Yes, already in use | Optimal for current scale |
| GIN tokens (current) | Yes, already in use | Efficient up to ~10M entities per type |

### Recommendation

**Don't add probabilistic structures for v1.0.** The current three-stage pipeline (anchors → tokens/embeddings → field scoring) is well-matched to the entity resolution problem at the scale EntityStore targets (thousands to low millions of entities per type). PostgreSQL's indexes handle the hot paths efficiently.

If a specific customer hits a scale bottleneck, the architecture supports adding MinHash or phonetic encoding as incremental enhancements without changing the core pipeline. The matching package's `EntityMatchConfig` already has extension points for new similarity functions and normalizers.

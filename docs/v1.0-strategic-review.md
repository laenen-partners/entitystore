## Simplify

### 1. Kill the `store` package leak

This is the single biggest DX problem. Consumers are forced to import two packages for basic operations:

```go
import (
    "github.com/laenen-partners/entitystore"          // for EntityStore
    "github.com/laenen-partners/entitystore/store"     // for BatchWriteOp, WriteActionCreate, TxStore
)
```

Every type consumers touch — `BatchWriteOp`, `WriteEntityOp`, `WriteAction`, `TraverseOpts`, `TxStore`, `WriteOpOption` — lives in `store/` and is re-exported via type aliases. But `Tx()` returns `*store.TxStore` directly, and generated `WriteOp` functions reference `store.WriteAction` and `store.WriteOpOption`.

**Fix:** The re-exports already exist. The remaining leaks are:
- `Tx()` returns `*store.TxStore` — wrap it in a root-package `TxStore` type
- Generated code imports `store` directly — change the codegen to reference root package types instead
- `WithTx(pgx.Tx)` is fine since `pgx` is a conscious dependency

This would let consumers write `entitystore.WriteActionCreate` instead of `store.WriteActionCreate` everywhere. One import, not two.

### 2. Collapse `GetEntitiesByType` and `GetEntitiesByTypeFiltered`

Two methods that do the same thing, one with an optional filter. Just make `GetEntitiesByType` accept the filter (nil = no filter). Same for any other method pair that differs only by an optional parameter. The filtered variant exists because SQLC generated two queries, but that's an implementation detail that shouldn't leak into the API.

### 3. `FindConnectedByType` has 7 parameters

```go
FindConnectedByType(ctx, entityID, entityType, relationTypes, filter, pageSize, cursor)
```

This is the ugliest signature in the API. It should take an options struct:

```go
type FindConnectedOpts struct {
    EntityType    string
    RelationTypes []string
    Filter        *QueryFilter
    PageSize      int32
    Cursor        *time.Time
}
FindConnectedByType(ctx, entityID, opts *FindConnectedOpts)
```

Same pattern as `TraverseOpts`. Consistent, extensible, readable at call sites.

### 4. `FindEntitiesByRelation` has no pagination

It's the only entity read method without a LIMIT. Add pageSize + cursor for consistency, or cap it internally at 1000 like `ConnectedEntities`.

### 5. 19 type aliases is too many

The root package re-exports 19 types from `store` and `matching`. This works but creates a confusing dual identity — `entitystore.StoredEntity` and `matching.StoredEntity` are the same type, and consumers mix both imports.

Can't eliminate these without breaking the `matching` package's independence (which is valuable). But document the convention: "import `entitystore` for consumer code, import `matching` only if building a custom Matcher or Embedder."

### 6. `Scoped` is a method, not a constructor pattern

```go
es.Scoped(cfg)  // returns *ScopedStore
```

This is fine for internal use but awkward for DI. If a service receives an `EntityStorer`, it can't scope it — the interface doesn't have `Scoped()`. This is correct (not all `EntityStorer` implementations would support scoping), but it means scoping must happen at construction time, before injection. Document this pattern explicitly.

## Clean up

### 7. Remove `matching.BuildAnchors`, `matching.BuildTokens`, `matching.TextToEmbed` from public API

These are the old JSON-reflection-based functions that the generated `Tokens()`, `EmbedText()`, and `Anchors()` replace. They still exist for the `Matcher` internals, but consumers should never call them directly anymore. Mark them as internal or move them to an unexported helper. At minimum, remove them from the examples and documentation.

### 8. The `codegen_example.go` is obsolete

It manually constructs `EntityMatchConfig` and `ExtractionSchema` to simulate what the codegen produces. Now that the codegen exists and is documented, this file teaches the wrong pattern. Replace it with an example that uses the generated functions (even if the generated code isn't checked in).

### 9. The `examples/store_example.go` is 730+ lines

Split into focused files: `example_write_test.go`, `example_query_test.go`, `example_traverse_test.go`, `example_scoped_test.go`. Make them `Example*` functions that `go test` can verify don't panic. Currently they're just `func` signatures that can't be validated.

### 10. ADRs have stale status

Several ADRs say "Proposed" but the features are shipped. Update to "Accepted" or "Implemented" with the version number.

## Expand

### 11. `godoc` is your README for library consumers

The package doc comment in `entitystore.go` is two sentences. For a v1.0, the godoc should be the primary onboarding surface — a paragraph explaining what EntityStore does, a minimal code example, and pointers to key types. Many Go developers discover libraries through pkg.go.dev, not GitHub READMEs.

### 12. One integration test that exercises the full pipeline

There are 87+ unit/integration tests, but no single test that does: create entity with codegen WriteOp → find by anchor → update with merge → traverse related → soft delete → verify invisible → hard delete → verify gone. This would serve as both a regression test and a living documentation of the happy path.

---

## Do NOT do

- **Don't add more query methods.** The API surface is already large (28 methods). Future query needs should go through `Traverse` or raw `WithTx` + pgx.
- **Don't add OpenTelemetry tracing.** slog is sufficient. OTel would add 5+ dependencies for something consumers can add themselves via middleware.
- **Don't build the SQLite backend** (ADR-004). Different deployment model, doubles the test matrix, zero demand signal.
- **Don't add batch-level options to `BatchWrite`.** The current `[]BatchWriteOp` signature is fine. If you need isolation levels or timeouts, use `WithTx` with a manually configured transaction.
